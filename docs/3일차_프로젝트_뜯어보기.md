## 1. 디렉터리 역할

### app/core/

프로젝트 전역에서 공통으로 사용하는 설정, 데이터베이스 연결, 공통 유틸리티를 두는 영역이다. 특정 API 기능 하나에만 속하지 않고 여러 기능에서 함께 사용하는 코드가 들어간다.

예를 들어 현재 프로젝트에서는 `app/core/config.py`에서 환경변수를 읽고, `app/core/db/databases.py`에서 DB 연결 엔진과 세션을 만든다.

### app/models/

데이터베이스 테이블과 매핑되는 SQLAlchemy ORM 모델을 작성하는 영역이다. ERD의 테이블 하나가 보통 Python 클래스 하나로 표현된다.

예를 들어 `users` 테이블은 `User` 클래스, `patients` 테이블은 `Patient` 클래스로 작성한다. 각 컬럼은 `mapped_column()`으로 정의하고, 테이블 간 관계는 `ForeignKey`로 표현한다.

### app/services/

비즈니스 로직을 작성하는 영역이다. API 계층은 요청을 받고 응답을 반환하는 역할에 집중하고, 실제 처리 규칙은 service 계층에 둔다.

예를 들어 회원 가입 시 이메일 중복 확인, 비밀번호 해싱, 권한 기본값 설정 같은 로직은 service에서 처리할 수 있다.

### app/apis/

FastAPI 라우터와 엔드포인트를 작성하는 영역이다. 클라이언트가 호출하는 URL, HTTP method, request body, response model을 정의한다.

예를 들어 `GET /practice_api/users` 같은 엔드포인트를 `APIRouter`로 작성하고, `app/main.py`에서 등록한다.

### pyproject.toml

Python 프로젝트의 메타데이터와 의존성을 관리하는 파일이다.

현재 프로젝트에는 FastAPI, SQLAlchemy, Alembic, asyncmy, pydantic-settings 등이 의존성으로 등록되어 있다. `uv sync`를 실행하면 이 파일과 `uv.lock`을 기준으로 필요한 패키지가 설치된다.

### uv.lock

설치된 패키지의 정확한 버전을 기록하는 잠금 파일이다.

`pyproject.toml`은 필요한 패키지 범위를 적는 파일이고, `uv.lock`은 실제로 어떤 버전이 설치되었는지 고정하는 파일이다. 팀원들이 같은 환경에서 개발할 수 있게 해준다.

## 3. DB 연결 구조

### app/core/db/databases.py

DB 연결에 필요한 엔진, 세션, Base를 정의한다.

현재 구조는 다음 흐름이다.

```text
1. app/core/config.py의 settings에서 DB 접속 정보를 읽는다.
2. mysql+asyncmy:// 형식의 DATABASE_URL을 만든다.
3. create_async_engine()으로 비동기 DB 엔진을 생성한다.
4. async_sessionmaker()로 DB 세션 팩토리를 만든다.
5. declarative_base()로 ORM 모델들이 상속받을 Base를 만든다.
6. async_get_db()로 API에서 사용할 DB 세션을 제공한다.
```

현재 DB URL 구조는 다음과 같다.

```python
DATABASE_PREFIX = "mysql+asyncmy://"
DATABASE_URI = f"{settings.DB_USER}:{settings.DB_PASSWORD}@{settings.DB_HOST}:{settings.DB_PORT}/{settings.DB_NAME}"
DATABASE_URL = f"{DATABASE_PREFIX}{DATABASE_URI}"
```

즉 `.env` 값이 다음과 같다면:

```text
DB_USER=ozcoding
DB_PASSWORD=pw1234
DB_HOST=localhost
DB_PORT=3306
DB_NAME=ai_health
```

실제 연결 주소는 다음과 같은 형태가 된다.

```text
mysql+asyncmy://ozcoding:pw1234@localhost:3306/ai_health
```

## 4. SQLAlchemy 모델 작성 흐름

SQLAlchemy ORM 모델은 ERD의 테이블을 Python 클래스로 옮기는 작업이다.

작성 흐름은 다음과 같다.

```text
1. ERD에서 테이블 이름을 확인한다.
2. app/models/ 하위에 테이블별 파일을 만든다.
3. Base를 상속받는 클래스를 작성한다.
4. __tablename__에 실제 테이블 이름을 적는다.
5. 각 컬럼을 Mapped와 mapped_column으로 정의한다.
6. PK, FK, unique, nullable, default 조건을 ERD에 맞게 작성한다.
7. app/models/__init__.py에서 작성한 모델을 import한다.
```

예시:

```python
from sqlalchemy import String
from sqlalchemy.orm import Mapped, mapped_column

from app.core.db.databases import Base


class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False)
    name: Mapped[str] = mapped_column(String(20), nullable=False)
```

ERD 기준으로 작성해야 하는 주요 테이블은 다음과 같다.

```text
users
patients
medical_records
xray_images
ai_analysis_results
```

테이블 간 관계는 `ForeignKey`로 표현한다.

예시:

```python
patient_id: Mapped[int] = mapped_column(
    ForeignKey("patients.id", ondelete="CASCADE"),
    nullable=False,
)
```

## 5. Alembic 마이그레이션 흐름

Alembic은 SQLAlchemy 모델 변경사항을 실제 DB 스키마에 반영하기 위한 도구이다.

전체 흐름은 다음과 같다.

```text
1. SQLAlchemy ORM 모델을 작성한다.
2. app/models/__init__.py에서 모델을 import한다.
3. Alembic이 Base.metadata를 읽어 모델 정보를 확인한다.
4. revision --autogenerate 명령어로 마이그레이션 파일을 생성한다.
5. 생성된 migration 파일을 확인한다.
6. upgrade head 명령어로 DB에 적용한다.
7. DB Viewer에서 테이블 생성 결과를 확인한다.
```

마이그레이션 파일 생성:

```bash
uv run alembic revision --autogenerate -m "create initial tables"
```

DB 반영:

```bash
uv run alembic upgrade head
```

주의할 점은 모델 파일을 만들기만 하면 Alembic이 자동으로 알지 못할 수 있다는 것이다. `alembic/env.py`에서 `from app import models`를 하고 있으므로, `app/models/__init__.py`에서 각 모델을 import해야 한다.

예시:

```python
from app.models.user import User
from app.models.patient import Patient
from app.models.medical_record import MedicalRecord
from app.models.xray_image import XrayImage
from app.models.ai_analysis_result import AIAnalysisResult
```

## 6. API 구현 흐름

FastAPI에서 API를 구현하는 일반적인 흐름은 다음과 같다.

```text
1. schemas에서 요청/응답 모델을 정의한다.
2. repositories에서 DB 조회/저장 코드를 작성한다.
3. services에서 비즈니스 로직을 작성한다.
4. apis에서 엔드포인트를 작성한다.
5. main.py에서 라우터를 등록한다.
6. Swagger UI에서 요청/응답을 테스트한다.
```

예를 들어 회원 목록 조회 API는 다음 흐름으로 동작한다.

```text
Client
→ GET /users 요청
→ app/apis/user_api.py의 endpoint 실행
→ service 호출
→ repository에서 DB 조회
→ schema에 맞게 응답 반환
```

현재 연습용 API처럼 간단한 경우에는 `apis` 안에서 리스트를 직접 사용해도 된다. 하지만 실제 DB 연동 프로젝트에서는 API, service, repository, schema, model의 역할을 분리하는 것이 유지보수에 좋다.

## 7. 정리

이번 프로젝트 구조는 FastAPI 애플리케이션을 계층별로 분리하기 위한 기본 구조이다.

```text
apis        요청/응답 endpoint
schemas     요청/응답 데이터 형식
services    핵심 처리 로직
repositories DB 접근 코드
models      DB 테이블 ORM 모델
core        공통 설정과 DB 연결
main.py     FastAPI 앱 시작점
```

SQLAlchemy 모델은 ERD를 코드로 옮기는 작업이고, Alembic은 그 모델 변경사항을 실제 DB에 적용하는 작업이다. 최종적으로는 모델 파일, 마이그레이션 파일, DB Viewer 확인 캡처가 함께 있어야 과제 완료 조건을 만족할 수 있다.
